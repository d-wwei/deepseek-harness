DeepSeek Harness / Agent V1.0 提案

从第一个 Agent 产品开始，构建 DeepSeek 原生 Harness 生态。如果 DeepSeek 的长期目标是 AGI，那么 Harness / Agent 不只是产品层应用，而是模型进入真实复杂任务、形成数据飞轮和开源生态的关键系统。

TL;DR · 一页提案

Takeaway：DeepSeek 做 Harness / Agent 的核心，不是复制一个 Coding Agent，而是从模型能力提供者，进入可验证、可治理、可回流的任务完成系统。

V1.0 可以概括为三件事：一个定位，一条护城河，一种生态姿态。

定位
SOTA cost-performance Agent

不以 Claude Code 的低成本替代为定位，而是在可验证工程任务上，用更低的「单位成功任务成本」逼近第一梯队完成率。

护城河
DeepSeek-native Harness

基于小样本 V4 API probe 与任务验证观察，它有一组通用 wrapper 会踩错的执行契约与成本结构，决定了它需要专属默认值，而不是接进一个通用外壳。

生态
社区邀请函：开源贡献回流 loop

把模型开源的传播势能，升级为围绕 DeepSeek 模型与 Harness 标准形成的开源 Agent 生态：官方搭骨架，社区补血肉，高价值反馈回流。

首个验证场景建议锚定 coding / engineering——它复杂度高、结果可验证、反馈密度高、用户愿意尝鲜，最适合先打穿复杂任务闭环。但它不是最终定义，通过 coding / engineering 能够抽象数字世界和知识工作所需的绝大多数能力，是一个高信号训练场。

第一部分

战略分析

Takeaway：DeepSeek 做 Harness / Agent 的核心，不是复制一个 Coding Agent，而是从模型能力提供者，进入可验证、可治理、可回流的任务完成系统。

1 · 背景为什么 DeepSeek 现在需要 Harness / Agent

Model + Harness = Agent。如果 DeepSeek 的长期目标是 AGI，Harness 就不是模型外的一层应用包装，而是把模型能力带入真实任务环境的执行、反馈、验证和风险控制系统。第一个 Agent 产品的意义，也不是复制 Claude Code / Codex，而是让 DeepSeek 掌握真实工作流、任务轨迹、失败反馈和开发者入口。

进一步看，DeepSeek 可以发挥自己的开源心智优势，把模型开源带来的开发者信任、全球传播和社区试验能力，升级为围绕 DeepSeek 模型与 Harness 标准形成的开源 Agent 生态。官方真正要控制的不是所有应用形态，而是生态骨架：让个人开发者、第三方生态开发者、开源项目、企业内部团队和云厂商围绕 DeepSeek 标准创造、连接、分发、评测，并把高价值反馈回流。

2 · 复杂任务与 AGI复杂任务是通往 AGI 的必经训练场和验证场

如果 AGI 不是抽象口号，而是可观察、可验证、可迭代的能力进步，它应该体现为 AI 能否在真实环境中持续完成越来越复杂、越来越长期、越来越开放的任务。不能简单说「能做复杂任务 = AGI」，但如果一个 AI 系统长期不能完成真实复杂任务，也很难被认为接近 AGI。

评测对象从「知道」转向「完成」

静态 benchmark 衡量模型知道什么；复杂任务衡量模型能否在现实约束中持续完成目标。它把能力问题从答题正确率，推进到长期目标、工具行动、环境反馈和结果验证。

能力尺度从正确率变成任务曲线

METR 的 task-completion horizon、SWE-bench、Terminal-Bench、GDPval 等趋势，都在追问同一件事：能不能完成、需要多少人工介入、结果是否可验证、单位成功任务成本是多少。

复杂任务不是 AGI，但不可缺席

复杂任务不能直接等同于 AGI；但真正有 AGI 含义的任务，通常会包含多步骤规划、长程保持、开放环境、工具使用、失败恢复、风险控制和跨场景迁移。

复杂任务必须落到 Harness

模型负责推理和行动意图，环境提供文件、代码库、终端、浏览器、文档和企业系统。Harness 把二者交互变成可调用、可观察、可验证、可治理的任务执行过程。

复杂任务执行闭环，Harness 要管理的六个环节：① 接入真实环境 → ② 调用工具与系统 → ③ 计划 / 行动 / 观察 → ④ 保持状态与上下文 → ⑤ 验证结果与证据 → ⑥ 管理权限与反馈。

3 · 不同模型需要不同 Harness复杂度由谁承担

Harness 不应是通用外壳。不同模型的上下文能力、reasoning 风格、工具调用稳定性、成本结构、速度和长任务表现都不同，因此不同厂商形成了不同的 Harness 信念。这里的差别不是谁的功能更多，而是复杂度由谁承担。

+展开三种 Harness 信念（Anthropic / OpenAI / Manus）与对 DeepSeek 的启示
方向性判断为什么会这样对 DeepSeek 的启示
Anthropic：极简 Harness + 简单 loop + 押注模型进化Claude Code 更相信强模型会逐步内化规划、纠错、工具选择、长程保持；Harness 重点是把上下文、规格、权限、session、hooks、skills、subagents 整理清楚，让模型在相对简单的 tool loop 里工作，可组合可调试但不过度工程化。轻、快、易懂，但长任务终止、失败恢复、成本预算、多轮验证更依赖模型自觉。DeepSeek 不能只套一层极简 loop 等模型自己进化，尤其 V4 的 thinking、tool_choice、cache、loop 终止都有明确工程约束。

Manus：任务经济性 + cache discipline应用公司视角：每轮调用、上下文复用、缓存命中、并发、失败重试和交付成本要在商业上成立，KV-cache discipline 与 cost-per-task 比单点能力展示更重要。DeepSeek 优势天然接近这一层（SOTA cost-performance、Flash / Pro 路由、长上下文），但成本纪律必须由 Harness 显式管理，不能留给用户或第三方 wrapper。

结论：DeepSeek-native Harness 不宜简单沿用 Anthropic 的极简 loop，也不应只复制 Codex 的产品形态，而是在事实标准兼容之上，形成围绕 V4 执行契约、成本结构和开源生态的工程控制面。

通用 wrapper 很容易在 max_tokens、thinking、tool_choice、cache、Flash / Pro 路由和 loop 终止这些点上静默浪费成本或直接出错。

3.2 DeepSeek V4 的工作性格画像

把 DeepSeek V4 当成一个工程师看，它更像是：诚实、勤奋、字面化、手快、成本效率高，但不会自己收手，也不替用户消解没说清的判断。Harness 的角色不是去改造模型本身的行为，而是给这位工程师配一个互补的项目经理。

勤奋但不收手

需要工具时会主动调用，不用幻觉答案替代真实查询；但持续失败时会一路查到硬上限。Harness 要自持步数、工具预算和 submit 收尾。

显式稳 / 隐含要写清

明确写出的 schema、验收标准、禁止项、停止条件，它通常遵守得很好；但「别动无关文件」「先问清楚再做」「冲突时听最新口径」这类默认规则，不能只靠它领会——Harness 要把隐含规则编译成显式约束、优先级和验收门。

行动优先于澄清

面对宽泛目标时倾向先动手，只有 Pro + thinking 更稳地先反问。Harness 必须强制先问 / 先 plan 的门。

近因偏好

遇到两条势均力敌的冲突约束，会可靠报冲突；若被迫取舍，倾向最后出现的约束。Context Engine 要在装配前去重、消解矛盾。

thinking 后外显计划变弱

开 thinking 后会把计划搬进隐藏的 reasoning，可见输出的结构会塌掉。Harness 必须自己持有 plan 状态，thinking 当升级而非默认。

诚实不装

能对照 spec 抓自己的问题，也会主动报冲突。这让 verifier lane、evidence contract 和 failure explanation 有更好的产品基础。

这些行为画像指向的不是 UI 层微调，而是 Harness 的职责边界：模型负责推理和行动意图，Harness 负责把约束、预算、状态、验证、权限和反馈变成可执行的工程机制。

3.3 所以：为什么必须 DeepSeek-native Harness

DeepSeek-native Harness 的必要性有两层。第一层是上一节的执行契约和成本结构——这些是机械事实，通用 wrapper 接 DeepSeek 会静默浪费成本或直接出错。第二层更根本：不同模型的工作性格不同，需要的 Harness 也不一样。同一个任务，配给一个「勤奋但不收手、遇歧义先动手」的工程师，和配给一个「保守、遇歧义先停下来问」的工程师，项目经理要做的事正好相反。DeepSeek V4 的性格恰好逼出一组特定的默认值——换一种性格的模型，下面这些就会反过来。

DeepSeek 的执行契约 + 工作性格 → 正确默认值（换个性格的模型会不同）[契约] 按难度升级到强模型        → Flash 为默认 + verifier lane（1/3 成本）
[契约] 默认开 thinking 求稳      → thinking 作升级策略，结构化输出走 non-thinking
[契约] thinking/output 独立预算   → 共享 max_tokens，分离管理防「思考吞答案」
[契约] 忽略缓存、随意拼接上下文     → stable-prefix 缓存，把成本结构当一等约束
[性格] 行动优先、遇歧义先动手     → 默认强制 plan / 澄清门（换个会先问的模型则放松）
[性格] 勤奋但不会自己收手        → 自持步数 / 工具预算 + submit 收尾
[性格] 近因偏好、隐含规则不自觉    → 装配前消解矛盾 + 把隐含规则编译成显式约束
[性格] 一思考就把计划搬进 reasoning → Harness 自持 plan 状态
[性格] 诚实、能对照 spec 自评     → 把 verifier / evidence 建立在它的诚实上

所以 Harness 不能是套在任何模型上的通用外壳。它的默认值，是「这个模型的执行契约 + 这个模型的工作性格」共同决定的——这正是它必须 native 的原因。

4 · Harness 的双重目标释放当前能力，并促进循环进化

Harness 的目标不是让 Agent「看起来能用」，而是提高可持续、可验证、可治理、低成本的复杂任务完成能力。它同时服务两个目标：先把当前模型能力释放出来，再让真实任务反馈反过来推动模型和 Harness 共同进化。

目标一
释放当前模型能力

模型本身提供智能，但复杂任务不是单次智能输出。Harness 要把 context、tools、loop、state、memory、verification 和 permission 组织起来，用强弱模型组合、多候选并行和验证并行，提高单位成本下的任务成功率。

目标二
促进模型与 Harness 循环进化

Harness 也是模型进化的数据接口。真实任务里的失败轨迹、工具调用、测试结果、用户修正、人工接管和最终验收，比静态问答数据更接近 AGI 所需的行动反馈，也能成为训练、评测和产品迭代的共同燃料。

5 · 用户采用曲线为什么现在是 Agent 产品的关键窗口

AI 能力增长很快，但用户采用不是同步发生。大多数用户仍停留在免费聊天、简单问答、内容生成、简单代码补全阶段；真正愿意持续把真实任务交给 Agent 的人仍是少数。这意味着机会不只在已有 coder 工具用户，而在下一层深度 Agent 使用者。

更长期的用户层次包括高强度开发者、开源 maintainer / startup engineer、企业内部工程团队、Pro knowledge worker 和组织级 workflow 用户。Coding-first 不是因为只有 coder 重要，而是因为 coder 是最适合先打穿复杂任务闭环的人群；V1.0 的 Harness 能力必须为更广泛的专业知识工作者和组织工作流留下扩展空间。

6 · 行业竞争格局不能只按模型层看

竞争不能只按模型层看。Agent 时代的竞争会沿着「能力来源 → 任务产品 → 用户入口 → 工作流入口 → 生态标准 → 企业控制」逐层展开。DeepSeek 如果只做一个同质化 Coding Agent，会同时面对国际 frontier labs、国内大厂入口、云厂商和新兴 Agent 产品的多层挤压。

结论：DeepSeek 的优势不在复制所有入口型产品，而在于用低成本模型 + 原生 Harness + 可验证任务系统 + 开源生态，建立一个可被多入口调用的任务执行层。

W · 开源不自动带来任务数据

模型被广泛使用，不等于官方拥有用户入口、真实任务轨迹和失败样本；第三方 Harness 的优化可能停在第三方产品里，无法回流官方模型与 Harness 设计。

O · 从模型开源升级到 Harness Ecosystem

发布 reference harness、SDK / runtime standard、task trace、eval case、skill / tool schema，让个人、开源项目、第三方 Agent、云厂商和企业团队围绕同一套标准创造价值。

T · 控制点可能外流

云厂商、大厂 IDE、开源 runtime 或第三方 framework 可能定义事实标准；若缺少 license、品牌边界、贡献回流和商业合作机制，生态越大，治理成本越高。

结论：开源若只停在模型层，会放大影响力但未必形成战略控制点；延伸到 harness standard / runtime / eval / trace / community contribution，DeepSeek 有机会从开源模型提供者升级为开源 Agent 生态的组织者。

8 · 战略总结SOTA cost-performance Agent + 开源生态 loop

如果 DeepSeek Agent 能在强公信力榜单上做到 SOTA，这是最强定位；更现实、也更符合 DeepSeek 结构性优势的目标，是 SOTA cost-performance Agent：在 SWE-bench Verified、Terminal-Bench、Aider Polyglot / 多语言任务和自建任务集上，以明显更低的 cost per verified task 接近第一梯队。

与此同时，DeepSeek 不能只思考「官方 App 如何抢入口」。如果官方产品之外的 Agent、云厂商、IDE 和企业内建平台承担了大量真实任务，而任务轨迹、失败样本和验证结果不回流，DeepSeek 就可能变成被广泛调用但不掌握进化飞轮的模型后端。因此需要建立 task trace、eval case、skill / tool schema、permission / sandbox / audit 抽象、贡献回流、云部署和品牌 / 商业合作边界。

外部战略窗口也在变化：非美国市场不只需要「最强模型」，也需要长期可获得、可控、可部署、可审计的智能能力。DeepSeek 的低成本强模型心智、开源心智、私有部署和原生 Harness，有机会把可调用模型升级为可获得、可部署、可治理的任务完成系统。（具体政策边界需以公开信源核实后再写入正式稿。）

9 · 战略控制点优先占位哪些关键环节

第一切入点应该任务价值高、结果可验证、反馈密度高、用户愿意尝鲜、能证明模型能力、能自然扩展并形成生态贡献。在这些标准下，coding / engineering 是很强的第一候选，但它不是最终定义，通过 coding / engineering 能够抽象数字世界和知识工作所需的绝大多数能力，是一个高信号训练场。

第一战场

Coding 复杂度高、验证相对清晰、反馈密度高、用户价值明确、可迁移性强，可以打磨 TaskSpec、Trace、Verifier、Hook、Context 和 Runtime。

铁杆用户

内部真实任务用户、高强度 Agent 用户、开源 maintainer、创业团队工程师、成本敏感但任务复杂的开发者和第三方 Harness 构建者。

优势任务

低成本多候选 / 多验证任务、长上下文可缓存任务、国内工程环境任务、私有化审计任务、environment hardening 任务、Flash workers + Pro planner / verifier 组合任务。

产品力

Agent-ready 项目（先检查依赖、测试、权限和上下文，让真实 repo 能被 Agent 接手）、低成本可验证执行（每次任务给出成本、验证证据和失败原因）、DeepSeek-native Router（按任务难度组合 Flash / Pro、thinking、上下文和 verifier，而不是让用户手动猜模型）、Context Health、Breakpoint Handoff、Trace / Eval / Skill Flywheel。

Task Runtime

前期很难要求用户整体迁移到一个新 Agent 产品，更现实的路线是让 DeepSeek 成为其他 Agent、CI、IDE、企业 workflow 可以调用的任务执行方，从既有工作流里切下更适合 DeepSeek 的任务。它把低成本强模型优势包装成可调用、可验证、可组合的工程任务能力。

回流协议

开源会让很多人基于 DeepSeek 做自己的 Agent。官方不应控制所有应用，但应控制 TaskSpec、TraceSpec、EvalSpec、SkillSpec、HookSpec、Runtime compatibility、Certified Runtime、Contribution / license / provenance——这决定 DeepSeek 是开源模型供应商，还是开源 Agent 生态组织者。

+展开面向其他 Agent 的最小任务闭环

V1.0 不需要完整开放平台，但至少要定义这套最小契约：

  • Capability Manifest — 能做什么、不能做什么、需要什么输入
  • TaskSpec — 结构化描述目标、上下文、约束、预算、权限、验收标准
  • Lifecycle API — submit / status / stream / cancel / resume / retry
  • Trace Stream — 机器可读的步骤、工具调用、成本、验证与失败
  • Artifact Contract — diff / 文件 / 报告 / PR / 日志 / 测试结果引用
  • Verdict / Evidence — 是否通过验证、证据、哪些验证被跳过
  • Typed Failure — 区分环境 / 权限 / 测试 / 上下文 / 预算 / 不确定等失败
  • Retry / Resume Semantics — 哪些可重试、应从哪个 checkpoint 恢复
  • Policy / Hook 接入 — 调用方传入预算、权限、安全与验收约束

  • 上层 Coding Agent
  • 多 Agent 编排器
  • CI / GitHub Bot
  • 企业 Workflow Agent
  • Benchmark / Eval Harness

9.8 假设验证

上面这些战略判断都是假设，不是结论。可以先设计可证伪的实验，把它们逐一降到可验证、可管理的风险。

验证定位

把 15 任务 benchmark 扩成程序级回归（多文件真 repo、100+ 任务），实测 cost per verified task 相对 Claude Code/Codex 的真实差距，找到确有优势的 2–3 类任务。

验证用户

招募 10–20 个高信号用户做 design partner，验证「是否愿意把下一个真实任务继续交给它」。

验证入口

选择 2–3 个上层 Agent / CI 集成方，验证「路由一部分任务给 DeepSeek Task Runtime」是否真降成本、是否愿意接 contract。

验证回流

验证一条「真实失败 trace → 脱敏 → eval case → 模型团队」的最小链路。

每条都设了「成立 / 不成立」的判据和对应动作——这决定哪些控制点优先投入、哪些先观望。

第二部分

产品方案（提案）

Takeaway：V1.0 是以真实工程任务为首个验证场景的 SOTA cost-performance Agent，通过 human-facing 与 agent-facing 双入口，交付 DeepSeek 原生 Harness 的第一个产品化入口。

1 · V1.0 产品定位三句话的组合

建议把 V1.0 定位成三句话的组合：SOTA cost-performance Coding Agent；靠谱友好的工程协作 Agent；面向用户和开源社区的第一封邀请函。它分别回答为什么值得试、为什么愿意继续用、为什么值得共建。

Benchmark · 值得试
SOTA cost-performance

在真实工程任务上，以更低 cost per verified successful task 达到接近第一梯队的任务完成率。关键不是 token 更便宜，而是同等预算下完成更多可验证任务。

Experience · 愿意用
可靠工程协作 Agent

像可靠工程师一样先理解目标，再规划、执行、验证、恢复和交付；不撒谎、不急着交活，能解释风险和不确定性。

Community · 愿意共建
Invitation / 邀请函

邀请用户把真实任务交给 DeepSeek Agent，也邀请开源社区基于官方 Core、Spec、Runtime、Trace、Eval、Hook 骨架共建 Harness 生态。

这里的成本单位值得说清：真正该比的不是「每百万 token 多少钱」，而是每个「已验证通过的任务」多少钱（cost per verified successful task）——一个便宜却常交付半成品的模型，真实成本反而更高。Harness 把 Verifier Lane 做成一等公民（deterministic 检查优先、不可判时才动用 Pro 当裁判），正是为了让 DeepSeek 的低成本兑现成低成本的可信成果，而不仅是低价 token。

之所以从 benchmark 出发：编程 Agent 是可以被外部评测强约束的领域——SWE-bench / Verified、Terminal-Bench、Aider Polyglot、SWE-rebench / Pro 等评测，约束了真实 GitHub issue 修复、端到端终端任务、多语言代码编辑、持续更新和任务成本。

而「Agent Ready」更适合作为用户侧表达：让真实项目进入 Agent-ready 状态，并用 DeepSeek 更低成本完成可验证工程任务。V1.0 不是普通 patch generator，而是 DeepSeek Harness 生态的第一版官方骨架。

2 · V1.0 的目标分解同时服务三类目标

V1.0 不是单纯发布一个工具，而是同时服务 DeepSeek、用户和开源生态：先把模型能力转化为真实任务完成能力，再把任务过程沉淀成可验证、可复用、可回流的 Harness 资产。

对 DeepSeek

建立 DeepSeek-native Harness 产品形态，拿到 trace、失败样本、人工修正和 eval case，形成 cost-performance benchmark 证明，并为 Desktop、IDE、SDK、Runtime、企业私有化打底。

对用户

让项目进入 Agent-ready 状态，完成第一个可验证任务；过程可见、可审阅、可暂停、可恢复，结果有验证证据，成功和失败都能沉淀为下次资产。

对生态

目标不是开源一个客户端，而是官方先搭好骨架，让社区贡献 skills、hooks、eval cases、environment recipes、routing policies、verification recipes、migration adapters 和 framework packs。

产品取舍

  • 先工程任务，后泛办公
  • 先可验证任务完成，后功能丰富度
  • 先同一 Core，后多入口扩展
  • 先兼容迁移，后自建生态壁垒
  • 先 review-gated 沉淀，后自动进化

第一版底线

宁可少做功能，也要把任务结果、验证证据、失败恢复和成本账做好；Memory / rule / skill / eval 都必须有来源、可审阅、可回滚，不能用「自进化」掩盖风险。

3 · 目标用户与需求洞察两类入口，同一个 Core

V1.0 的目标用户不是「所有写代码的人」，而是愿意把真实工程任务交给 Agent 的高信号调用方。之所以分成人类用户和 Agent 用户，是因为真实任务有两种进入路径：人类把目标、项目上下文、信任判断和最终责任交给 DeepSeek；上层 Agent、CI、IDE、workflow 则把结构化子任务、预算、权限和验收条件交给 DeepSeek Runtime。

前者决定产品能不能获得真实用户和高质量反馈，后者决定 DeepSeek 能不能进入更大的任务网络、成为可复用的执行层。

更进一步：人类用户真正购买的不是「自动写代码」，而是把复杂工程任务托付出去时的信任感、控制感和可接管性；Agent 用户要的也不是「会聊天的开发者」，而是一个低成本、可验证、可编排、失败语义清楚的工程任务原语。

+人类用户：把 Agent 当工程伙伴的人

早期画像：Agent-heavy developer、开源 maintainer、创业团队工程师 / 技术负责人、企业内部开发者、国内工程环境用户和非美国市场开发者。
触发场景：真实 repo 的 bug 修复、跨文件改造、环境硬化、测试补齐、依赖迁移、长任务推进，以及需要边做边解释的工程探索。
核心焦虑：怕 Agent 假装完成、乱改无关文件、看不懂过程、中断后重来、权限越界，或把未经审核的规则和记忆污染项目。
产品必须满足：先澄清目标和风险再执行；过程有 timeline / 计划 / diff / 验证报告 / 成本提示；可暂停、可恢复、可接管，失败也能解释并沉淀；memory / rule / skill 必须 review-gated。

+Agent 用户：把 DeepSeek 当任务执行组件的系统

早期画像：上层 Coding Agent、多 Agent 编排器、CI / GitHub Bot、IDE / Editor Agent、企业 Workflow Agent、Benchmark / Eval Harness、云厂商和第三方 Agent 平台。
触发场景：把低成本可验证的子任务路由给 DeepSeek——批量修复、测试生成、迁移适配、verifier lane、失败复现、eval case 生成和环境检查。
核心焦虑：怕能力边界不清、任务状态不可读、失败只返回自然语言、无法重试恢复、成本不可控，或 runtime 违反上层系统的权限策略。
产品必须满足：Capability manifest 与 TaskSpec / schema validation；Lifecycle API、event stream、trace、artifact contract；verdict / evidence、typed failure、retry / resume；budget / cost accounting、policy / hook 与权限结果。

Agent Core 与 Task Runtime 的区别

Agent Core 更像任务智能中枢：理解目标、澄清约束、生成计划、组织 context、选择工具、执行 agent loop，做 Pro / Flash、thinking / non-thinking、reasoning effort、max_tokens、task budget 路由，并根据 observe / test / error / user feedback 修正路径。Task Runtime 更像系统控制面：创建、暂停、恢复、取消、重试、并发、超时；管理 sandbox、worktree、terminal、browser、权限、审批、预算、审计；对外提供 TaskSpec、Lifecycle API、Trace Stream、Artifact Contract、Verdict / Evidence。

人类入口与 Agent 入口

Human-facing Surface 第一版先交付一种形态（桌面端或 CLI/TUI 皆可，由团队按目标用户定），支持 init、task 创建、plan 确认、权限审批、trace 查看、diff / artifact / verification 审阅。Agent-facing Surface 先做 Minimal Task Runtime：submit / status / stream / cancel / resume / artifact / verdict。不同入口复用同一套 state，底层不被任何单一入口锁死。

兼容层与壁垒层

V1.0 既要降低迁移成本，也要建立 DeepSeek-native 的长期优势——兼容层让人愿意进来，壁垒层让任务越跑越适合 DeepSeek。

兼容层

让用户和第三方系统愿意进来：兼容 API、tool calling、MCP、skills、hooks、A2A，并提供 Claude / Codex / OpenCode 迁移路径。

壁垒层

让任务越跑越适合 DeepSeek：Router、Context Health、Verifier Lane、Constraint & Hook、Environment Hardening、Trace / Eval / Skill Flywheel，把成本、长上下文、开源心智和 V4 行为特征变成任务完成优势——默认值全部回指 §3 实测。

+展开 V1.0 架构边界（必须打通 / 可后置）

必须打通：至少一种人类入口 Surface 调用同一 Runtime；Minimal Task Runtime API 可被其他 Agent / CI 调用；TaskSpec、TraceSpec、Artifact Contract、Verdict / Evidence Contract 初版；session、task state、trace、artifact、verification、memory / skill / eval candidate 共享同一套存储和来源记录；hook / permission / budget / audit 从第一版进入 Runtime。

6 · 能力核心能力 × 特色能力

从一个朴素的问题倒推：用户凭什么敢把真实任务交给它？答案分两层——核心能力是「靠谱工程师」的基本盘，特色能力才是 DeepSeek 才有、且别人不容易抄的差异化（每条都标了它的依据）。

核心能力 · 让人敢托付的基本盘

可信人格（行为契约）

做不到会直说，没验证过不假装搞定；交付时讲清楚哪些测过了、哪些没测、哪些得你来拍板。

优秀的工程师思维（工程判断与规划）

先弄懂你到底要什么、会牵动哪些地方、有没有更稳的做法，而不是挑最省事的地方改一刀就交。

自主工作流编排

它能自己往前推、自己编排步骤，但有步数和预算上限、不会无限跑下去；你随时看得到它在干嘛、花了多少、要不要批准危险操作，也随时能喊停。

自主低成本验证

跑测试、lint、构建、看 diff 自己验证，不靠嘴上说「好了」；还能用 DeepSeek 的低成本同时跑几个候选、再用验证器挑出对的——同样的预算，办成更多被验证过的任务。

特色能力 · DeepSeek 才有、且难被复制

实测
原生模型路由

不是一遇到难题就上最贵的模型，而是按「这活儿要花多少钱、风险多大、有没有自动验证兜底」来分配 Pro/Flash 和思考力度。实测里 Flash 在有测试兜底时和 Pro 一样能干完，只花 1/3 的钱。

实测
思考预算托管

该深想的深想、该直给的直给，这些档位由 Harness 替你管，不甩给你一堆参数；还专门防一个坑——DeepSeek 想太多会把预算用光、最后返回一个空答案。

实测
上下文健康管理

1M 窗口不是往里灌料：稳定的部分缓存下来省钱、关键规则钉在模型记得住的位置、压缩时绝不丢掉验收标准和红线。

工程方法
环境硬化

顺手把环境、依赖、测试、回滚准备补齐，让这个项目以后交给任何 Agent 都更省事，而不是改完就走人。

工程方法
断点中继与热恢复

任务中断、上下文被压缩、甚至换个入口继续，都能从上次断点热恢复接着干——多数 Agent 一崩溃 / 一压缩就丢状态、得从头再来，这是把「长任务不重来」做成产品力。

工程方法
沉淀经验，skill 受控进化

好用的做法自动沉淀成可复用 skill、踩过的坑沉淀成回归测试，让 Agent 逐步学会最佳实践；但每条都 review-gated、有来源、可回滚——是「受控进化」，不是放任它自己乱长。

工程方法
记忆自动整理

后台自动整理项目背景、你的偏好、过去决定和任务历史，形成可读可编辑的记忆图谱；区分个人 / 项目 / 组织记忆，你随时看得到、改得了、删得掉——记忆是资产，不是黑盒。

双入口
Agent 和人类双入口

不强求你换掉现在的工具——你的 Claude Code、CI、流水线可以把一部分活儿外包给 DeepSeek 跑，人类也能直接用。既降低迁移门槛，也让 DeepSeek 拿到更多真实任务。

国内场景
国内工程环境适配

国产依赖源、镜像、企业内网、飞书钉钉、私有化部署这些国际产品照顾不到的场景，它原生适配。

7 · 情感化设计人类入口的 owned relationship

Agent-facing 入口需要的是 stable contract；human-facing 入口需要的是 owned relationship。人类用户不是只在调用能力，而是在决定是否把真实任务托付给 DeepSeek、是否允许它记住自己的工作方式、是否长期回来、是否愿意把一次成功协作讲给别人听。

所以情感化设计不应被理解为「把 Agent 做可爱」，而是降低托付门槛、建立可持续协作关系、让长期记忆变得可感知，并把可靠任务完成转化为用户愿意分享的产品记忆点。更进一步，DeepSeek 可以在日常协作中逐渐形成可审阅的 persona.md：记录用户角色、经历、工作方式、偏好、禁忌和协作历史，用它持续优化与人类用户的交流方式。

理论依据

Norman
情感化设计影响用户是否愿意开始、继续和回忆一次体验

Donald Norman 在《Emotional Design》中把体验分为本能层、行为层和反思层。对 Agent 产品来说，行为层仍是任务可靠性；但反思层决定用户是否觉得「这是我的长期协作对象」，而不是一次性命令行工具。

SDT
人类用户需要自主感、胜任感和关系感

Self-Determination Theory 可以解释为什么命名、偏好控制和共同里程碑有用：用户要能控制 Agent 的沟通方式，要感到自己和 Agent 一起完成了更难的任务，也要感到这种协作关系不是冷冰冰的外包。

Fogg
情感化入口降低第一次真实托付的启动阻力

Fogg Behavior Model 里，行为发生需要动机、能力和提示同时成立。DeepSeek 的模型能力提供 ability；小鲸鱼、工作状态提示和里程碑提示提供 prompt；真实完成、可恢复、可分享则强化 motivation。

CASA
用户会自然把社会性投射到会对话、会行动的系统上

Media Equation / CASA 研究提醒我们，人会不自觉地用社会规则回应计算机。Agent 越像协作者，越需要主动设计身份、边界和可控性，否则用户会自行脑补关系和能力边界，反而产生虚假信任。

Social
社交货币来自「我用它完成了什么」，不是来自 mascot 本身

Jonah Berger 在《Contagious》中强调 social currency。对 DeepSeek Agent 来说，值得分享的不是「我有一只可爱小鲸鱼」，而是「我的项目进入 Agent-ready 状态」「这个真实任务被低成本验证完成」「我教会了一个可复用 skill」。

八个设计点

身份
领取自己的 DeepSeek Agent

第一次启动时让用户领取并命名自己的 DeepSeek 小鲸鱼，也可以换成更专业的 workspace identity。这个动作的意义不是卖萌，而是把「我要不要试一个工具」转化为「我要不要建立一个长期协作入口」。

Soul
定义性格、表达习惯和协作偏好

用户可以设置它更直接、更严谨、更少打扰，或更像一个会解释风险的 senior engineer。偏好包括称呼、语气、解释密度、主动追问阈值、风险提醒强度和复盘方式；所有偏好都应可查看、可修改、可删除。

Persona
逐步形成用户的 persona.md

在日常协作中，Agent 沉淀用户画像：用户是谁、做过什么项目、常用技术栈、偏好的沟通节奏、容易焦虑或反感的点、对风险和解释的偏好、共同经历过的成功和失败。它不是一次性问卷，而是从长期任务、复盘、用户修正和显式确认中逐步形成。persona.md 应作为 review-gated memory：每次新增重要判断都让用户可见、可改、可删；它不能给用户贴不可更改的心理标签，也不把 MBTI、星座、生辰八字等轻量材料用于严肃决策，唯一目标是让 DeepSeek 更懂如何与这个用户协作。

互动
在协作过程中加入轻量 buddy / snippet

Agent 在合适节点给出短反馈：开始规划、发现风险、测试通过、需要审批、失败但已定位原因。它不应打断主流程，而应像工作旁白一样帮助用户理解进度、风险和下一步。

状态
把 Agent 形象用于工作状态可视化

小鲸鱼根据任务状态变化，而不是随机表演情绪：阅读代码、调用工具、等待审批、运行测试、验证失败、恢复中、准备交付。用户看到的是可理解的任务生命周期，不是不可解释的拟人化。

仪式
把长期协作变成可感知的里程碑

记录第一次真实任务成功、第一次失败后恢复、第一次用户教会规则、共同完成第 10 / 100 个任务、项目首次达到 Agent-ready。里程碑不是游戏化装饰，而是让用户感知「我和 DeepSeek 在一起变强」。

分享
让用户获得可分享的社交货币

支持生成脱敏的 Verified Task Card、Agent Readiness Report、Project Harness Score、开源贡献卡和月度协作回顾。分享对象不是私人代码和聊天记录，而是任务结果、验证证据、节省成本和可复用贡献。

边界：情感化必须服务于托付、验证和长期协作，不能抢过任务可靠性，也不能制造「Agent 会承担责任」的错觉。企业版默认提供更专业的 workspace initialization，并允许管理员关闭命名、形象和主动惊喜，只保留可审计的偏好、状态和协作设置。

8 · 成功指标测什么，以及怎么测

指标不能太多。V1.0 只围绕三个问题：任务是否真的完成？用户是否愿意继续托付？生态是否开始回流？

北极星指标
单位成本下的可验证真实任务完成数

它同时约束三件事：任务必须真实、结果必须可验证、成本必须可计算。若第一场景是编程 / 工程任务，可用 SWE-bench、SWE-rebench、Terminal-Bench、Aider Polyglot 和 DeepSeek Task Suite 做参照。三个一级指标：

任务完成质量

完成率、首次完成率、验证通过率、失败恢复成功率。

用户托付意愿

连续任务数、继续使用率、计划 / 交付接受比例、人工接管和回滚比例。

生态回流质量

任务转 eval、实践转 skill、失败转回归测试、贡献被复用。

观察指标只做诊断：平均任务耗时、单次任务成本、模型路由升级比例、上下文压缩触发次数、权限拦截次数、任务恢复次数、其他智能体调用成功率——帮助诊断，但不和北极星并列。

8.4 怎么测：埋点、实验、定性、研究员闭环

埋点

每个任务发结构化 trace，所有指标从 trace 计算，不靠主观打分。trace 即真相源——这也是本方案采用的验证方法。

实验

离线用冻结、去污染、留出的内部 SWE-style 任务集做 A/B（已有 15 任务真 pytest 雏形）；在线策略变更走灰度。

统计纪律

micro-eval 找方向、冻结大集做决定；报告给区间与置信度，诚实标注未覆盖范围。

定性

铁杆用户访谈 + diary study；任务后「托付意愿」问卷；failure taxonomy 每周复盘。

研究员闭环

失败 trace → eval case → 训练信号，产品团队负责这条数据契约，每周 eval triage。模型与 Harness 的共同进化，落地形态就是这条管道。

第三部分

开源社区回流 loop

Takeaway：DeepSeek 的开源优势不能只停留在模型权重，应该升级为围绕 Harness 骨架、真实任务和社区贡献形成的生态回流 loop。

1 · 为什么V1.0 必须考虑开源生态

DeepSeek 的开源优势既是机会，也是风险。

机会

用户信任更高、社区愿意贡献、可私有化、可审计、可部署，并且更容易形成国际开发者生态。

风险

第三方可以基于模型做更快的 Harness；官方若没有标准，社区贡献会碎片化；云厂商或平台也可能拿走用户入口和任务 trace。

所以，开源社区不是 V1.0 之后的运营活动，而是产品设计的一部分。V1.0 要从一开始就思考：哪些东西开放，哪些进入标准，哪些贡献能回流模型与 Harness，哪些品牌、数据和商业边界必须保护。

2 · 开源目标官方骨架 + 社区血肉

开源目标不应是「把所有东西都开源」，也不是让社区各自做一套松散插件，而是官方先搭好可长期演进的生态骨架，再让社区在这个骨架上贡献真实场景、任务能力和改进反馈。

官方定义共同语言

TaskSpec、任务生命周期、权限与预算边界、trace、artifact、evidence、verdict、skill、hook、recipe、eval，以及版本、署名、来源、审核、安全扫描和淘汰机制。

社区贡献场景能力

工程环境包、任务技能包、验证与 eval 包、生态适配包，把 DeepSeek Agent 带进官方内部覆盖不到的真实任务场景。

官方建立回流路径

社区贡献经过自动测试、安全扫描和 benchmark 验证后进入 curated registry、官方推荐能力、默认策略或评测集，并反哺 Harness 设计、模型 eval 和训练反馈。

3 · 第一版开源边界能跑起来的最小骨架

第一版不要试图一次性开源完整生态。更现实的边界是：开源一个能跑起来的最小 Agent 骨架，开放少量稳定贡献入口，用真实任务验证哪些接口值得继续固化。建议 V1.0 只覆盖三类内容：

可运行入口

能在本地 repo 跑真实任务，形态不限。

最小协议

TaskSpec v0、trace v0、artifact / evidence 最小格式。

少量示例

starter skills / recipes、环境硬化示例、eval case 示例和贡献指南。

第一版暂不承诺：完整 marketplace、大规模 benchmark harness、全量 MCP / A2A / Claude / Codex 迁移适配、丰富语言 / 框架 / 企业系统 packs、云端 runtime、企业治理后台、高风险 hook 能力和商业 telemetry 聚合。 这样的边界更像「邀请函」：可信、可运行、可贡献。

4 · 生态回流机制让个人实践变成官方资产

仅有「邀请函」的姿态不够。真正的回流机制，是让社区里的个人实践有机会变成官方生态资产——四步闭环：

1 · 贡献入口

真实失败案例、可复用 skill、环境配置方法、验证方法、迁移适配经验都可成为候选贡献。

2 · 筛选标准

可复现、可脱敏、来源和协议清楚、对一类任务有复用价值、不引入明显安全风险。

3 · 分层吸收

community examples → curated registry → official feedback。

4 · 反馈与激励

贡献者必须看到自己的贡献到了哪里、产生了什么影响、是否被采纳。

重点不是「收集尽可能多数据」，而是建立一条可信通路。

而这条可信通路必须有一道明确的贡献闸门（Contribution Gate）：脱敏、授权与 license 核验、PII / secret 扫描、reasoning 默认不上传、企业数据本地留存、以及抽样人工复核。否则这条回流在战略上很漂亮，却可能无法通过安全与法务审查——让飞轮转得起来的前提，是它先合规可控。

5 · Co-builder 激励用八角模型设计可持续的贡献动机

不能只靠 DeepSeek 品牌好感，也不能把激励理解成 badge、排行榜、token 额度这些表层手段。这里用 八角模型（Octalysis）* 的八大驱动力，系统地设计贡献者的长期动机：

Epic Meaning & Calling
史诗意义与使命感

让贡献者清楚自己在参与一件更大的事：共建一个开放、低成本、可私有化的 Agent 基础设施，而不是给某个产品写插件。

Development & Accomplishment
进步与成就感

一条看得见的成长路：user → contributor → trusted contributor → maintainer → co-builder partner，每升一级对应更大的权限和话语权。

Empowerment of Creativity
创意授权与反馈

把工具给齐（spec、starter 模板、eval harness、本地调试），让人能快速做出 skill / recipe 并马上看到效果、拿到反馈。

Ownership & Possession
所有权与拥有感

署名、来源、命名空间、版本和维护权都归贡献者；以后真有 marketplace，收益归属也提前讲清楚。

Social Influence
社交影响与关联

好贡献被看见、被用上：curated registry、官方案例、公开致谢、benchmark 引用、社区 spotlight。

Scarcity & Impatience
稀缺与渴望

早期 co-builder 名额、实验功能抢先体验、官方联调 office hour 这些「限量」机会，留给早期高强度贡献者。

Unpredictability & Curiosity
未知与好奇

不定期的 bounty / 挑战任务，加上「你的贡献会走到哪一层」的开放结果——可能进官方默认策略、甚至成为模型训练信号。

Loss & Avoidance
损失规避

把贡献者最怕的几件事消除：清晰 license、脱敏规则、兼容承诺、弃用周期、安全审核、争议处理——让人敢押长期资产，不怕「贡献被无署名吃掉、接口很快废弃」。

* 八角模型（Octalysis）：游戏化设计专家周郁凱（Yu-kai Chou）提出的行为分析框架，把驱动人持续投入的动机归纳为八大核心驱动力。这里用它来系统设计开源贡献者的长期激励，而不是只堆 badge 和排行榜。

6 · 开源协议与治理边界分层协议，三类不混

开源协议不是法务附录，而是生态回流、商业边界、云厂商合作和品牌信任的一部分。入口层尽量开放，核心层保留治理和回流空间，企业与云服务层保留商业合作边界。代码协议、数据协议、品牌协议不能混在一起——代码贡献、任务样本、运行轨迹、skill / plugin 贡献、DeepSeek compatible / certified 标识，都需要不同授权与治理规则。

7 · 国内与国际社区一个全球社区，两类运营侧重

DeepSeek 不应把中文社区和英文社区做成两套割裂系统，也不应把国内定位成「反馈场」、把国际定位成「信任背书场」。更合理的思路是以 GitHub / 开放文档 / 公开协议作为全球共同底座，国内外都看作真实使用、生态贡献和产品反馈来源——只是场景密度不同：

国内社区

更快暴露中文工程环境、国内依赖和镜像、企业内网、飞书 / 钉钉 / 企业微信、私有化部署、国产云和国产软硬件适配问题。

国际社区

带来全球开源项目、多语言多框架、公开 benchmark、协议互操作、MCP / A2A / Claude / Codex / Cursor 迁移和跨区域协作经验。

V1.0 至少要有三个最小动作：一个公开可信的项目入口（双语 README、quickstart、contribution guide、roadmap、governance）、一个可回流的反馈机制（高价值反馈进入公开 issue、RFC、eval case 或 curated registry）、一组双语示例任务。

第四部分

长期展望及风险边界

Takeaway：DeepSeek 不需要一开始重定义 Agent 标准，更现实的路径是先兼容事实标准，再用 DeepSeek-specific optimization profile 证明适配价值。

1 · 长期展望在事实标准之上建立 DeepSeek 优化层

长期看，DeepSeek 不应假设自己能重新定义 Agent 行业标准。OpenAI / Anthropic 的 API、tool use、message format，以及 MCP、A2A 等协议和生态，已经在很多开发者、云厂商、IDE 和 Agent 平台里形成事实标准。更现实的方向不是「请外部平台适配 DeepSeek 标准」，而是先兼容已有事实标准，再提供 DeepSeek-specific optimization profile，用更低成本、更高成功率或更好稳定性证明适配价值。

Layer 1 兼容兼容 OpenAI / Anthropic API 格式、主流 tool use、MCP、A2A 和已有 skill / plugin 生态，降低第三方迁移成本。
Layer 2 profile提供 Pro / Flash / thinking 路由、context cache、long-context health、低成本多候选、verifier lane、max_tokens / thinking budget 管理、trace / eval 回流格式。
Layer 3 证明用 reference implementation、examples、benchmarks、compatibility tests 证明：同一 Agent 框架下按 profile 接入 DeepSeek，可以在部分任务获得更好的 cost、reliability 或 latency。
Layer 4 合作面向云厂商、IDE、Agent 平台和企业内部团队，提供联调支持、参考部署、迁移适配和可选 compatibility 标识。

标准影响力来自采用，不是发布。

2 · 云厂商与企业私有化能否进入真实组织工作流

云厂商和企业私有化不是简单渠道合作问题，而是 DeepSeek 能否进入真实组织工作流的问题。云厂商大概率会做跨模型 Agent 平台，企业内部团队也更关心部署、安全、审计和成本，不会天然围绕某一个 DeepSeek 官方 Agent 重新搭系统。

中国市场更复杂：腾讯云、阿里云、华为云不是中立渠道——它们既有云基础设施，也有自己的模型、Agent 产品和企业客户关系，更可能把 DeepSeek 放进自己的跨模型平台，而不是把入口和反馈交给 DeepSeek。

因此 DeepSeek 的目标不是让云厂商或企业绑定官方 Agent，而是让他们在自己的平台里更容易、更便宜、更安全地把 DeepSeek 用好；同时在客户授权和脱敏前提下，设计 eval / failure case / routing result 的回流路径。

竞争型云

腾讯、阿里、华为——重点是兼容、性能证明和客户需求牵引，不应期待它们把 DeepSeek 官方 Agent 作为主入口。

合作型渠道

运营商、行业云、集成商——重点是参考部署、私有化方案、行业场景包和运维支持，帮它们补齐 Agent 产品能力。

企业直连

大型企业内部平台团队——重点是可审计、可控成本、可接入内网，让 DeepSeek 成为企业跨模型体系里的高性价比任务执行选项。

企业私有化的核心购买理由不是「模型更强」，而是代码数据不出域、权限审批审计可控、接入内网、成本可预测、可复盘。V1.0 不做完整 enterprise console，但要预留 TaskSpec、Policy / Hook、Trace / Artifact、Eval / Failure case、Deployment recipe。

3 · 商业化机会围绕成功任务成本收费

商业化不是 V1.0 的主轴，但商业化边界要从第一版设计清楚。否则产品容易走向两个极端：只卖 API token，价值太薄；或者过早商业化社区贡献和用户任务数据，伤害开源信任。DeepSeek Harness / Agent 的商业化核心，不应是「开源 Core 收费」，也不应只是「模型 token 收费」，而应围绕真实任务完成收费。更准确的价值单位是成功任务成本：模型成本 + Harness 执行成本 + 验证成本 + 人工介入成本 + 失败恢复成本。

个人开发者 Pro

卖给高强度个人开发者

托管 sandbox / cloud workspace、后台长任务和任务队列、多候选并行、verifier lane、断点恢复、trace / artifact / session 云端同步，把模型、执行环境、验证和存储打包成任务额度。

团队工作空间

卖给研发团队

团队共享 session、项目记忆和 skill，repo 级 policy，PR / issue / CI 工作流集成，成本预算、任务用量、成功率和失败原因报表，团队级 eval / regression suite。

企业与私有化

卖给不能用公有云 Agent 的组织

VPC / 私有云 / 本地部署，SSO / RBAC / 审计日志 / 数据留存，内网代码库、知识库、工单系统连接器，私有 skill registry、私有 eval、私有 trace store。

生态能力包

长期有想象力，但不能写成已验证的成熟商业模式；V1 更适合定位为生态基础设施和增长资产，先让社区贡献被使用、被验证、被吸收，而不是直接售卖。

进入顺序

开源 CLI / Core 先获真实任务 → 个人开发者 Pro → 团队工作空间 → 企业与私有化部署 → 更长期再探索生态能力包、企业专业包、marketplace 和贡献者分成。

不应优先做

大而全订阅制、基础 Core 和协议收费化、认证收费、黑箱 trace / skill / eval 数据资产、把生态能力包写成已验证 marketplace、过度依赖云厂商渠道。

4 · 主要风险如何避免走偏

风险部分不只是负面清单，而是说明 DeepSeek 如何避免走偏。这里把 V1.0 最容易偏航的风险收束为六类。

coding 太窄

不要做成 Claude Code / Codex 的低价替代品。应对：把 coding 定义为数字任务世界的首个高信号训练场，并沉淀可迁移的 TaskSpec、trace、eval、skill、hook、memory 和 policy。

能力不足同质化

短期 V4 绝对能力可能不及第一梯队，只「更便宜 + 功能相似」没迁移动力。应对：不承诺第一版绝对 SOTA，而是在部分任务上形成 cost-performance frontier，把低成本转化为多候选并行、verifier lane、失败恢复、Safe Parallel Workspace 和任务级预算控制。

第三方抢占控制点

第三方基于 DeepSeek 做更快 Harness。应对：官方定义 spec 和 reference implementation，建立 co-builder 回流机制，用 benchmark suite、optimization profile 和 compatibility tests 形成适配优势。

云厂商截流

企业经云渠道采购，用户关系与 trace 不回流。应对：设计 DeepSeek-specific optimization profile、compatibility tests、trace / eval return protocol，并保留官方开发者社区和开源入口。

数据隐私安全

Agent 接触代码、密钥、配置和企业数据。应对：local-first、secrets guard、protected paths、sandbox、audit log、permission policy、redaction、user-reviewed contribution。

Agent 面过大

V1.0 把 Agent-facing 做成完整平台会失焦。应对：只做 Minimal Task Runtime，把完整 SDK、marketplace、enterprise runtime 后置，先服务 CI、GitHub Bot、上层 Agent worker 等高信号集成。

5 · 边界声明

V1.0 应明确不做泛办公万能 Agent，不用功能数量证明价值，不把 1M context 当作单纯堆叠上下文窗口的卖点，不把 memory / skill 写成自动自进化，不承诺第一版完整 open platform，不把情感化体验放在任务可靠性之前，也不把 coding 写成 DeepSeek Agent 的唯一未来。

6 · 支撑材料

候选 Idea、范围清单、证据索引和后续待验证问题不进入主线。完整论证、完整 SWOT、竞争格局，以及对 DeepSeek V4 的实测原始数据（probe 脚本与任务验证记录）可作为支撑材料保留，不进入主线。

结尾留给讨论的三个开放问题

前面四个部分已经给出完整的判断与方案。如果进入正式讨论，最值得先对齐的是下面三个问题——它们直接决定 V1.0 的方向取舍：

首个场景

DeepSeek 内部对「首个高信号任务场景」的判断是什么？是否就是 coding / engineering？

Harness 优先级

希望 Harness 优先服务模型训练反馈（数据飞轮），还是优先形成用户入口（产品增长）？

开放边界

如果第一版要开源，哪些部分适合开放，哪些部分必须保留为官方控制面？

这份方案想做的，是把模型、产品、工程、社区和真实任务反馈放进同一张图里思考——它既是一份判断，也是一个可以继续讨论、迭代和落地的起点。
